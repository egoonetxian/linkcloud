DRDS实例的性能瓶颈分析

用户使用DRDS提供的IP、端口以及账号密码进行操作的时候即仅能操作在DRDS上创建的数据库以及数据表。在mysql客户端（或代码驱动）产生的请求会首先到达DRDS服务器，DRDS即会对用户请求的sql进行解析并且根据分库策略对解析后的请求分发到对应的RDS上的分库上，然后RDS根据解析出的结果在对应的分库上执行sql得到结果经过DRDS服务器反馈至客户端。
从上面的访问方式可以查看到DRDS实例的请求其实是受两方面产品的规格限制的。首先是DRDS的配置，DRDS是分配对应的CPU和内存给用户，以保证用户DRDS可以处理用户请求的QPS。该QPS指DRDS服务器可以处理的请求频次，如下图：

其次，DRDS的请求还需要受RDS的性能配置影响，因为所有的sql请求最终其实还是在RDS上执行，因此需要保证在能够满足DRDS的QPS的限制的同时还要满足RDS在内存/cpu/iops等限制。
需要特别注意的一点是：由于DRDS到后端的RDS上的请求都是对其中不同的分库的连接，很导致RDS的连接数增长很多，特别是当一台RDS上存在太多的DRDS的分库的时候，因此需要尽量避免这种情况发生。

DRDS出现3009错误的解决方案

用户在通过DRDS执行sql的时候，有时由于SQL执行的时间较长会出现如下错误：

com.mysql.jdbc.exceptions.jdbc4.MySQLNonTransientConnectionException: Communications link failure
The last packet successfully received from the server was 30,024 milliseconds ago. The last packet sent successfully to the server was 30,024 milliseconds ago.

从报错信息上可以查看到是由于用户的慢查询超过了DRDS的超时时间的错误。由于DRDS本身是一个分布式中间件产品，对于网络延迟有较敏感的产品，因此DRDS会会对超时时间有严格的设置。而用户的有时的需求就是通过DRDS执行一个大的OLAP操作，此时用户可以通过设置hint来调整DRDS到RDS之间的超时时间。如下面示例：

/TDDL:socket_timeout=10000/select count(*) from tddl5_users;
socket_timeout的单位是毫秒，如果socket_timeout=0，则该SQL不会超时。
另外用户也可以通过hint来设置DRDS的执行方法和执行策略，详细的请参考：https://docs.aliyun.com/?spm=5176.2020520107.103.11.yAkf0M#/pub/drds/user-guide/user_drds_hint。

DRDS是否支持分布式join以及复杂SQL如何支持

不是所有的join都能够支持，例如对于大表之间的join，就因为执行代价过高，速度过慢而被我们限制，但大部分的join的语义，我们可以帮助用户以最高效的方式完成。
我们处理join的原则非常简单：尽可能让join发生在单机。

1. 按照同一个维度进行切分：如果我们能够让join物理上发生在单台机器上，那么任何一类的复杂查询都是可以直接支持的。一般而言，这就意味着参与join的多张表按照同一个维度进行切分。例如，一个用户有多个商品，每个商品都有自己的商品特征。这时候，如果您需要join,可以将所有数据按照用户，或者按照商品进行切分，那么join就会物理上的发生在同一个机器上，我们能够很轻松的保证所有在单机发生的join查询，物理上都能够查出数据，对于非常复杂的SQL，我们也允许您通过注释的方式，直接告知我们切分条件，这样就可以绕开我们的SQL解析器进行查询。
2. 小表复制：同时可以有选择的将一些不经常更新的，数据量比较小的元数据表复制到全部的节点上。这样，大表join小表的时候，就从一个分布式join变为了本地join，当然，这种过程需要一定的代价，也就是元数据表内的数据更新，可能在一小段时间(50~100ms)后才能在分库内看到。
3. 在线查询与离线查询分离：对于复杂的大表和大表的分析和统计类查询，我们比较推荐您采取专门的分析引擎来获取报表数据，比如使用ODPS等，这类查询使用传统数据库架构，在数据量非常巨大的时候，很可能会影响线上的应用，因此我们比较建议您能将在线和离线查询分开。

您可以通过console来自己尝试各类切分条件和join模型，来帮助您更快的了解我们能做到什么，不能做到什么。
